Purchasing Dashboard

How we track every part we buy — and how vendor emails turn themselves into orders. Architecture & design review.

amur-supplychain.web.appJuly 2026
The one-liner

A dashboard that keeps itself up to date

It tracks what we need to buy (BOM), what we've ordered, and our vendors. Its standout feature: an email robot that reads vendor emails automatically and turns them into pre-filled order cards — so orders stop being typed in by hand.

The rule that governs everything: the AI only ever proposes. A human approves before anything becomes real.

The big picture

How it fits together

The core principle

AI proposes.
People decide.

Nothing the robot produces is ever committed automatically. Every parsed email lands in a Review Inbox as a suggestion. A human approves it — one click — before it becomes a real order.

This is deliberate friction. It's the safety gate that lets us trust automation without trusting it blindly.

End to end

How an order flows in

  • Vendor emails a quote → someone CC's purchasing@ (or forwards it)
  • Within 10 min the robot parses it, maps parts to the BOM, checks for duplicates
  • A review card appears in the dashboard (or is suppressed/flagged if it's a duplicate)
  • A human reviews, edits if needed, clicks Approve → it becomes a real order
  • Later emails (shipped, tracking, delayed) → update cards → approve → status advances
Two intake paths

Every order enters one of two ways

1 · Official PO (Brex)

  • Download the PO PDF, drag it onto the form
  • A deterministic parser reads the fixed Brex layout
  • No AI — fast, exact, free

2 · Quote / CC / website order

  • No PO exists — the email thread is the record
  • CC purchasing@ → the robot makes a review card
  • AI, because vendor formats vary
The differentiator

The email robot

  • Runs inside a dedicated mailbox (purchasing-bot@), polls every 10 minutes
  • Sends each email + PDF to Claude, gets back structured data (schema-enforced)
  • Maps line items to our BOM part numbers; classifies quote / order / shipping update
  • Writes only suggestion cards — it can never touch real orders, BOM, or vendors
  • The API key lives only in the script — never in the browser, never in the repo
Keeping it clean

Duplicate detection

Before a card appears, the robot checks it against existing orders by identity — PO number → vendor's order number → fuzzy match:

  • Certain duplicate (re-sent order) → suppressed, logged, never clutters review
  • Real update (shipping / tracking / delay) → surfaced as an "Order Update"
  • Uncertain (same vendor + total) → shown, flagged "possible duplicate"
  • New → normal card

Runs in plain code, not AI — scales to thousands of orders at zero extra cost.

Security posture

Protected — with known, tracked gaps

✓ In place

  • Database locked to company Google accounts
  • No secrets in the page (and scrubbed from git history)
  • Outsiders get "Access denied"
  • Robot write-scoped to review cards only
  • XSS verified clean · full audit trail

⚑ Deferred (with triggers)

  • No auto-backups yet (Spark plan) — fix first
  • No per-role access — any teammate can edit all
  • Bot's DB access broader than its behavior
  • App Check not enabled
Why we cut the corners we cut

The trade-off principle

Defer what's cheap to add later and low risk now.
Invest early in what's irreversible: losing data, and committing bad data.

  • Deferred: auto-backups, role separation, App Check, scoped bot access — each with a named trigger for when we fix it
  • Invested up front: the human-approval gate, and a full security hardening — because those are the catastrophic-if-wrong ones

Every deferral is a tracked decision with an upgrade path — not debt left to rot.

Cost

What it costs to run

  • Hosting + database: effectively $0 at this scale (free tier)
  • The only spend is the robot's AI calls: ~$0.10–0.15 per email
  • Production estimate: ~$30–100/month depending on volume
  • A spend cap makes the worst case bounded — no surprise bills
