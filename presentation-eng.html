Why build, not buy?

What earns a custom app over a spreadsheet / ERP

A Sheet handles the tracking (and gives collaboration + versioning free). An ERP is premature at this scale. We built a thin app because the value is a workflow a spreadsheet can't do:

  • AI email → structured review → human-approve pipeline — dedup, BOM-matching, a staging area you vet before anything commits. Not a grid; an application.
  • Data governance — validation, computed-not-stored stats, soft-delete/restore/audit. A shared sheet degrades to chaos at 10–15 editors.
  • A foundation for automation — check-in agents, quote/PO parsing, shipment tracking.

Honest scope: for pure tracking a Sheet would suffice. This is a deliberate bet on the workflow + roadmap — shipped fast and cheap, with a known graduation path (right).

Stack: static single-page React on Firebase Hosting · Cloud Firestore (rules-enforced) · Google Apps Script → Claude API (key server-side only) · pdf.js for PO parsing. No servers to run — cheap, fast to ship. Trade-off: the front-end is currently a bundled artifact (migration planned, last slide).

Trust & safety model

AI proposes · humans commit

  • AI output never auto-commits — everything lands in a review queue a person approves
  • The robot's DB write is scoped to review cards only — it cannot mutate orders/BOM/vendors
  • Claude API key lives only server-side (Apps Script), never in the browser or repo
  • Every order gets a unique id; dedup collapses/flags duplicates deterministically (in code, scales)
Security posture & deliberate trade-offs

Hardened, with tracked deferrals

✓ In place

  • Domain-locked Firestore + deny-all fallback
  • No secrets in client; git history scrubbed
  • Access-denied gate · XSS verified · audit log
  • Automatic backups (PITR 7-day rewind)

⚑ Deferred (triggered)

  • No per-role access (trusted team; recoverable)
  • Bot IAM broader than behavior (code-scoped)
  • App Check off · repo→org pending

Principle: defer what's cheap-to-add & low-risk-now; invest early in the irreversible (data loss, bad commits). Every deferral has a fix trigger.

Reliability / how bugs are handled

Caught early, can't lose data

  • Automated logic tests (5 suites, 40+ checks over the real reducers) — a regression test per bug; run before deploy
  • Post-deploy smoke test — serves 200 · shell rendered · no data leak · security gates hold
  • Recovery net — PITR + Activity-log restore + Export JSON (proven in a real incident)
  • Playbook — contain (revert) → root-cause → fix → regression test → verify. Code + data both reversible.

Deferred: full browser E2E (Playwright + Firebase emulator, fake auth) — lands with the source migration.

Where it's going

Graduation path

Bridge → maintainable

  • Rebuild the bundled front-end as a source Vite/React project — in parallel, same Firestore, zero-downtime cutover
  • Per-entity Firestore docs → true multi-editor concurrency
  • Reduce bus-factor: a second maintainer

Pre-wider-rollout hardening

  • App Check (monitor → enforce)
  • Bot account lockdown · scoped IAM role
  • Repo private + 2FA + company org

Honest framing: it's a well-scoped MVP/bridge — cheap, shipped, secured, tested — that de-risks and teaches us what a permanent system (or a bought tool) needs.
