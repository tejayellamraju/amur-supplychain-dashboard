Why we built it — build vs. buy

The decision

Need: track procurement for Amur 002, and stop hand-entering orders from vendor emails.

Options considered

  • Google Sheet — great at tracking; collaboration + version history free. Weak at a controlled email→approve workflow and data integrity at 10–15 editors.
  • ERP (Cofactr/Fulcrum) — right at production scale; cost + overhead premature at 5–15 people.
  • Thin custom app — what we chose.

Why custom

  • The value is a workflow, not a table: email → AI-parsed → dedup/BOM-match → review card → human approve.
  • Structured data + computed stats a stray cell edit can't break.
  • A base for later automation (chasers, quote/PO parsing).

For pure tracking a Sheet would have done. We built for the workflow — and kept it thin.

Architecture · 2 of 3 — components & choices

The stack, and why

Front-end

  • React SPA, static-hosted on Firebase Hosting. Serverless — nothing to run. Deliberate: fast/cheap to ship. Trade-off: one bundled file today.

Data

  • Cloud Firestore, single doc, realtime listeners. Security rules are the access boundary. Stats computed at render, not stored.

Automation

  • Google Apps Script in a dedicated bot account — timer-triggered, no infra to host. Reads Gmail, writes Firestore.

AI & parsing

  • Claude API for email extraction — structured-JSON output, key server-side only.
  • pdf.js for Brex POs — deterministic parse, no AI (fixed layout).

Everything is managed/serverless — no boxes to run, patch, or scale. The cost is a bundled front-end (migration planned).

Keyed by order identity, not thread — so one thread can yield several orders, and one order across threads collapses to one card. Matching runs in code (scales, zero AI cost); the AI only extracts.

Data model & trust boundaries

Where data & access live

  • One Firestore doc (dashboard/main) holds all arrays; coverage/spend computed at render, never stored.
  • Access: rules allow read+write only for verified @fourier.earth/fourierearth.com; deny-all fallback. Client gate is UX; rules are the boundary.
  • Robot runs in a dedicated bot account; its write is masked to pendingOrders only — cannot touch orders/BOM/vendors.
  • Secrets: Claude key only in Apps Script; never in the browser or repo (git history scrubbed).
  • Trust model: AI never auto-commits — everything lands in a review queue a human approves.
Trade-offs & known limitations

What we punted on, and why

  • Single-doc writes → last-write-wins on concurrent edits. Fine at current scale; recoverable. Fix later: per-entity documents.
  • Front-end is a bundled artifact — hard to maintain, bus-factor of one. Source migration planned.
  • No per-role access — any verified user can edit/delete all. Accepted for a small trusted team; backups + audit cover it.
  • Bot IAM broader than its behavior, App Check off, repo not yet in org — each deferred with a trigger.

Rule we followed: spend early on what's irreversible (data loss, bad writes); defer what's cheap to add and low-risk now — and tag each deferral with when we do it.

Reliability & a real incident

Caught early, can't lose data

  • Automated tests — 5 suites, 40+ checks over the real reducers; a regression test per bug; run before deploy.
  • Post-deploy smoke test + PITR backups (7-day rewind) + Activity-log restore.

Incident: approving duplicate review cards reused one order id → a single delete cascaded across the duplicates. Root cause: approve reused the card's id. Fix: unique id per approved order + a regression test. Recovered via soft-delete + PITR — zero data lost.

Deferred: full browser E2E (Playwright + Firebase emulator, fake auth) — lands with the source migration.

Open questions & roadmap

Where I want your input

Open questions

  • When to move single-doc → per-entity Firestore? What write-contention threshold?
  • Auth for a future in-dashboard AI assistant — Cloud Function vs. token?
  • Keep building this, or fold into an ERP as scale grows?

Roadmap

  • Rebuild front-end as source (parallel, same Firestore, zero-downtime cutover)
  • Per-entity docs for multi-editor concurrency
  • App Check · scoped bot IAM · repo → org

It's an MVP, on purpose. We rebuild it or buy a tool when we outgrow it — and by then we'll know what to ask for.
